## 0. 目标与范围

**这一份要回答的问题**：当用户进入"循环生成章节"的稳态阶段后，**怎么保证长程一致性**——

- 5 章前埋的种子，第 8 章还能记得 / 还能回收
- 用户的"题材共鸣·我不爱什么"在第 20 章不会因为 prompt 长度爆炸而丢失
- 角色行为在支线里不会突然 OOC（Out Of Character）
- 用户回档到第 6 章，从第 7 章开始重新生成，**前面的伏笔不能被遗忘**

**不回答的问题**（留给后续文档）：
- 7 件产物的精确字段定义（02 只给出"用在哪"，03 给 schema）
- 多 agent 协作（双视角的导演/演员分工）——单线版本做完再谈
- 评测方法（怎么测"一致性好不好"）——这是 04 的事

**核心思路（蕾姆酱的判断）**：

> 一致性问题 = **状态保持问题**。LLM 单次上下文有限，不能把整个故事塞进 prompt。
> 解法 = **状态分层 + 摘要压缩 + 种子加权**。
> 状态 = 7 件产物（世界树/风格宪法/题材共鸣/主线/支线/人物/种子）。
> 分层 = 三层 prompt（基座层永远在 / 动态调参层按需拼 / 最近轮次层滚动）。
> 摘要 = 阶段摘要 + 章末极简摘要（前者压缩信息，后者定位节点）。
> 加权 = 种子按"距当前段数"算权重，过期的不再占 prompt 槽位。

## 1. 三层 prompt 结构（骨架）

> 这是 02 文档的**核心**。所有一致性机制最终都落到 prompt 的三个槽位里。

### 1.1 三层定义

| 层 | 内容 | 更新频率 | 长度预算 | 不可压缩 |
|---|---|---|---|---|
| **基座层** | 风格宪法 + 题材共鸣 + 世界树基座 | 几乎不变 | ~800 token | 是（删了风格会飘） |
| **动态调参层** | 当前种子权重清单 + 当前阶段摘要 + 当前密度配比 | 每章重算 | ~1200 token | 否（按需裁剪） |
| **最近轮次层** | 最近 N 章极简摘要 + 最近 1 章全文 | 滚动更新 | ~2000 token | 否（滑窗） |

**总预算 ~4000 token**（基座 + 动态 + 最近），余量给 LLM 输出 3000 字章节用。

### 1.2 各层详细

#### 1.2.1 基座层（永远在 prompt 头部）

```yaml
# 不变项：进入故事时定一次，跨回档不丢
style_charter: |
  <风格宪法，~300 token>
  例：「散文式 / 冷叙述 / 不滥情 / 留白 / 心理活动不超过 3 句/段」
genre_resonance: |
  <题材共鸣：用户接受什么 + 拒绝什么，~200 token>
  例：「接受：日常细节、克制的情感、暗线
       拒绝：爽文、过度巧合、廉价感动」
world_base: |
  <世界树基座：时间线/地理/核心设定，~300 token>
  例：「现代都市，2019 年夏，杭州，主角 28 岁程序员，妻是高中语文老师」
```

**关键约束**：
- **回档不丢**（基座是故事真正的"地基"）
- **基座修改 ≠ 回档**（B 解读：基座改了，老枝叶保留但不再被回溯到）
- token 预算硬上限 800，超出必须裁剪（裁剪顺序：world_base > style_charter > genre_resonance）

#### 1.2.2 动态调参层（每章重算）

```yaml
# 变动项：每章生成前 1 步重新计算
active_seeds: |
  <当前权重 ≥ 0.3 的种子，~400 token>
  例：「种子 #3（1987 年的收音机）- 0.7 - 上次出现：第 5 段
       种子 #7（父亲的信）- 0.4 - 上次出现：第 12 段」
stage_summary: |
  <最近阶段摘要，~500 token>
  例：「第 6 阶段（3-7 章）：主角发现父亲遗物，开始与妻疏远...」
density_config: |
  <当前密度配比，~300 token>
  例：「具体性 1.2 / 主观性 0.8 / 密度 0.6 / 题材共鸣 1.0」
  （基于 v3.1 模型 + LLM-as-judge 反馈动态调整）
```

**关键约束**：
- **每次章节生成前重算**（不是缓存）
- 种子清单按权重排序，取到 token 上限为止
- 阶段摘要按"距当前段数"分桶（见 §3）

#### 1.2.3 最近轮次层（滚动窗口）

```yaml
# 滑窗：保证最近记忆清晰
chapter_summaries_recent: |
  <最近 N 章极简摘要，~1500 token>
  例：「第 12 章：主角整理父亲遗物，发现 1987 年收音机
       第 13 章：妻询问，主角回避
       第 14 章：主角独自听收音机，回想少年时
       ...」
last_chapter_full: |
  <最近 1 章全文，~500 token 或 1000 字截断>
```

**关键约束**：
- N 的值动态（不是固定的 3 章或 5 章）——看 token 预算够不够塞下
- **章末极简摘要的格式**见 §4
- 最后 1 章**保留全文**（因为是"上一次"的完整状态）

### 1.3 三层拼接顺序

```
[System Prompt]
  ├─ 基座层（800t）
  ├─ 动态调参层（1200t）
  └─ 最近轮次层（2000t）
[User Prompt]
  └─ 当前用户输入（"我期望 xxx" 或 "我 xxx"）
[Assistant 输出]
  └─ 新章节（~3000 字）
```

**为什么基座在 System、动态在中间、最近在最后？**
- System 区放"长期约束"（LLM 训练时这个位置权重最高）
- 最近轮次靠近用户输入（LLM 注意力机制对上下文末尾响应最强）
- 动态调参夹中间，作为"承上启下"

### 1.4 三层与回档的关系

| 行为 | 基座层 | 动态调参层 | 最近轮次层 |
|------|--------|------------|------------|
| **进入故事** | 初始化 | 空 | 空 |
| **生成第 1 章** | 写入 | 写入空种子清单 | 写入本章摘要 |
| **生成第 5 章** | 不变 | 重算种子权重 | 滚动更新 |
| **回档到第 3 章** | 不变 | 重算（基于第 3 章状态） | 截断到第 3 章之前 |
| **基座修改** | 重写 | 重算 | **保留**（老枝叶不动） |

**关键不变量**：基座层一旦初始化，**几乎不动**（用户主动改才动）；动态层是**唯一每章重算**的；最近层是**唯一滚动更新**的。

## 2. 种子加权算法

> 种子 = 伏笔（按 01 文档 1.5 节合并概念）。本节定义**怎么决定哪些种子该出现在 prompt 里**。

### 2.1 种子属性（埋下时标记）

每个种子埋下时，**必须标记 4 个属性**（不是 1 个，是 4 个）：

```yaml
seed_id: 3
content: "1987 年的收音机"  # 种子本体的具体描述
planted_at_segment: 7      # 埋下时的段号
importance: 主线推进        # 1️⃣ 重要性：影响主线推进 / 支线故事 / 小巧思
size: 中线                 # 2️⃣ 种子大小：计划展开成多大的故事线
planned_interval: 10      # 3️⃣ 计划揭露间隔：5/10/20 段（设计意图）
orientation: 主角成长      # 4️⃣ 种子导向：主角成长 / 关键成员关系 / 剧情翻转 / ...

# 下面是程序维护字段（不需用户填）
last_seen_segment: 12
weight: 0.0                # 按 §2.2 公式重算
status: planted            # planted / resonating / harvested / expired
```

**4 个属性都是 enum**，**填值**在 §2.3 / 2.4 / 2.5 表里查。

### 2.2 权重公式（v0.1 草稿，4 维度加权求和）

```
weight = overdue_score × 0.4
       + importance_score × 0.25
       + size_score × 0.2
       + orientation_score × 0.15
```

每个维度独立计算，结果都在 0-1 区间，**加权求和后理论范围 0-1**。

| 维度 | 占比 | 性质 | 来源 |
|------|------|------|------|
| **1. 超出值 (overdue)** | 0.40 | 连续值（按段数算） | §2.3 双重缩放 |
| **2. 重要性 (importance)** | 0.25 | enum 查表 | §2.4 |
| **3. 种子大小 (size)** | 0.20 | enum 查表 | §2.5 |
| **4. 种子导向 (orientation)** | 0.15 | enum 查表 | §2.6 |

**为什么 0.4 最高**（欧尼酱原文）：计划揭露间隔是**设计意图**，到了该揭露的时间**越拖越紧迫**——这是种子系统最核心的驱动力。

### 2.3 维度 1 · 超出值（双重缩放 + sigmoid）

**计算公式**：

```
overdue = max(0, current_segment - planted_segment - planned_interval)
normalized_overdue = overdue / sqrt(planned_interval)   # 双重缩放：用 sqrt 软归一化
overdue_score = sigmoid(normalized_overdue × k)         # k 是斜率参数（v0.1 取 k=2）
```

**为什么用 `sqrt(planned_interval)` 而不是直接 `overdue`**：
- 短间隔种子（planned=5）超期 1 段应该敏感
- 长间隔种子（planned=30）超期 1 段不该敏感
- `sqrt(5) ≈ 2.24` vs `sqrt(30) ≈ 5.48` —— 长间隔种子的"1 段"被压低
- **比线性比率 `(overdue/planned)` 平滑**（比率对短间隔种子过激）

**sigmoid 函数**：`sigmoid(x) = 1 / (1 + e^(-x))`，把结果限到 0-1，**k=2 让曲线在 x=0-2 区间有显著上升**。

**示例**（k=2）：

| 种子 | planned | overdue 段数 | normalized | overdue_score |
|------|---------|-------------|-----------|---------------|
| #3（收音机） | 10 | 5 | 5/√10 = 1.58 | sigmoid(3.16) ≈ 0.96 |
| #7（父亲的信） | 5 | 2 | 2/√5 = 0.89 | sigmoid(1.79) ≈ 0.86 |
| #2（已过期） | 20 | 0 | 0 | sigmoid(0) = 0.5 |
| #5（待埋） | 15 | -3 | 0 (max) | sigmoid(0) = 0.5 |

**蕾姆酱的设计直觉**：
- **未到揭露时间**（overdue=0）→ 该维度贡献 0.5（中性），让其他维度决定排序
- **刚超期** → 快速上升
- **严重超期**（超 1 个 √planned）→ 接近 1.0

**待欧尼酱拍**：
- k=2 是否合适？（k 越大曲线越陡，k=1 更平缓）
- 是否需要给 overdue_score 加一个**下限**（比如超期 0 段时设为 0 而不是 0.5）？

### 2.4 维度 2 · 重要性（预设数值表）

| importance enum | importance_score | 说明 |
|----------------|------------------|------|
| **主线推进** | 1.0 | 不揭露故事走不下去 |
| **支线故事** | 0.6 | 支线张力来源 |
| **小巧思** | 0.2 | 锦上添花 |

**分档逻辑**：1.0 / 0.6 / 0.2，**间隔 0.4 留出"重要性之间"的中庸分值**（如果未来要加 enum）。

### 2.5 维度 3 · 种子大小（预设数值表）

| size enum | size_score | 说明 |
|-----------|------------|------|
| **长线** | 1.0 | 贯穿全文（10+ 段）|
| **中线** | 0.6 | 一到两个阶段（5-10 段）|
| **点状** | 0.2 | 单次出现即可（1-3 段）|

### 2.6 维度 4 · 种子导向（预设数值表）

| orientation enum | orientation_score | 说明 |
|-----------------|-------------------|------|
| **剧情翻转** | 1.0 | 颠覆读者/玩家预期 |
| **关键成员关系** | 0.8 | 主配关系变化、敌友反转 |
| **主角成长** | 0.6 | 主角内心/能力变化 |
| **支线揭示** | 0.4 | 解锁背景/伏笔 |
| **小巧思** | 0.2 | 装饰性细节 |
| **氛围营造** | 0.1 | 场景/情绪烘托，无剧情推进 |

**分档逻辑**：
- **剧情动力源**（剧情翻转 / 关键成员关系 / 主角成长）→ 0.6-1.0
- **辅助修饰**（支线揭示 / 小巧思 / 氛围营造）→ 0.1-0.4
- 这个分档是**蕾姆酱的直觉**，欧尼酱可调整

### 2.7 完整权重计算示例

**假设三个种子，current_segment = 17**：

| 种子 | 重要性 | 大小 | planned | 埋下段 | overdue | overdue_score | imp | size | orient | **weight** |
|------|--------|------|---------|--------|---------|---------------|-----|------|--------|-----------|
| #3「收音机」 | 主线 (1.0) | 中线 (0.6) | 10 | 7 | 0 | 0.5 | 1.0 | 0.6 | 0.6 | 0.5×0.4 + 1.0×0.25 + 0.6×0.2 + 0.6×0.15 = **0.71** |
| #7「父亲的信」 | 主线 (1.0) | 长线 (1.0) | 5 | 13 | -1 | 0.5 | 1.0 | 1.0 | 0.6 | 0.5×0.4 + 1.0×0.25 + 1.0×0.2 + 0.6×0.15 = **0.74** |
| #5「街角早餐店」 | 支线 (0.6) | 点状 (0.2) | 3 | 15 | -1 | 0.5 | 0.6 | 0.2 | 0.2 | 0.5×0.4 + 0.6×0.25 + 0.2×0.2 + 0.2×0.15 = **0.41** |

**解读**：
- 种子 #7 权重最高 0.74 —— 虽然还没超期，但**长线 + 主线 + 重要导向**让它靠前
- 种子 #3 权重 0.71 —— 刚到 planned 间隔，**已超期 0 段**这一维度贡献 0.5 中性
- 种子 #5 权重 0.41 —— 点状 + 支线 + 装饰导向，**优先级低**

**待欧尼酱拍**：
- k=2 的斜率是否合适
- 未超期时 overdue_score=0.5 是否合理（还是设为 0 让其他维度绝对主导）

### 2.8 阈值与 prompt 行为

| 权重区间 | 行为 |
|---------|------|
| **≥ 0.7** | **强制进 prompt**（基座层 + 动态层都提示） |
| **0.3 - 0.7** | **可选用**（按 token 预算塞，塞不下就丢弃） |
| **< 0.3** | **不进 prompt**（但仍在种子表里，万一未来要回溯） |

**为什么 0.3 和 0.7 两个阈值**：
- 0.7 是"必须"（剧情张力需要）
- 0.3 是"可以"（给 prompt 留弹性）
- 中间段是 LLM 的判断空间

**注意**：上一版 §2.2 的 0.7/0.3 阈值**仍然适用**——但**权重公式改了**（从 base × recency × importance 改为 4 维加权求和），**新公式下"什么算高权重"需要重新校准**。建议保留 0.7/0.3 阈值，**但需要实跑验证**。

### 2.9 prompt 注入格式

**在动态调参层的 `active_seeds` 字段**：

```markdown
## 当前活跃种子（按权重降序）

1. **种子 #3「1987 年的收音机」** [权重 0.7]
   - 埋于第 7 段，最后出现第 12 段
   - 状态：resonating（已强化 2 次）
   - 提示：本章可考虑再次出现或回收

2. **种子 #7「父亲的信」** [权重 0.49]
   - 埋于第 13 段，未再出现
   - 状态：planted
   - 提示：距埋下 1 段，建议本章强化一次

(剩余种子因 token 预算已截断，参考种子表)
```

**关键设计**：
- **不告诉 LLM "必须用"**（让 LLM 觉得种子是"建议"不是"命令"，降低 OOC 风险）
- 给 LLM 留**判断空间**（"可考虑" "建议"）
- 截断时**保留权重高的**（按降序填到 token 上限）

## 3. 阶段摘要机制（v0.1 暂用 15 章，远期需重新设计）

> 阶段摘要 = 多个章节的压缩版，比章末极简摘要更粗粒度。**当章节数变多时**，光靠"最近 N 章摘要"会爆 token。

### 3.1 触发与分层

| 层级 | 触发 | 写什么 | 长度 |
|------|------|--------|------|
| **章末极简摘要** | 每章生成完 | 1-3 句关键事件 + 种子变动 | ~80-120 token |
| **阶段摘要** | 每 15 章（或剧情节点切换时） | 5-8 句 + 人物状态快照 | ~500 token |
| **整故事摘要** | 第 30 章 / 第 60 章 / 故事收尾 | 15-20 句 + 主线进度 | ~1000 token |

**为什么是 15 章？** 蕾姆酱的判断（待欧尼酱拍）：
- 3000 字 × 15 章 = 45000 字，~15000 token（远超 LLM 上下文）
- 500 token 摘要能压缩到 3%，足够回溯
- 15 章 = 用户大约 2-3 周的阅读量（按每天 1-2 章估）

### 3.2 章末极简摘要的字段

```yaml
chapter_id: 14
chapter_range: "第 14 章（段 38-42）"
key_events:
  - "主角整理父亲遗物，发现 1987 年收音机"
  - "妻询问，主角回避"
  - "主角独自听收音机，回想少年时"
seed_changes:
  - planted: #7「父亲的信」
  - resonating: #3「1987 年的收音机」（强化 1 次）
character_state:
  protagonist: "疏离、回避"
  wife: "关切、疑惑"
unresolved:
  - "父亲为何留这封信"
  - "妻是否知道父亲的事"
```

**字段设计逻辑**：
- **key_events**：3 件以内（多了就是章节复述）
- **seed_changes**：结构化记录（不用 LLM 算）
- **character_state**：每章 1-2 个词，**避免累积成小作文**
- **unresolved**：存疑点（驱动后续剧情）

### 3.3 阶段摘要的字段

```yaml
stage_id: 4
stage_range: "第 6-20 章（段 21-60）"
stage_name: "遗物之谜"  # 可选，LLM 命名
narrative_arc: |
  主角因父亲遗物开始与妻疏远，
  逐步发现父亲不为人知的过去，
  妻察觉后开始追问。
main_threads:
  - thread: #3「收音机」
    progress: 0.4
    note: "已强化 2 次，未回收"
  - thread: #7「父亲的信」
    progress: 0.1
    note: "刚埋下"
character_arcs:
  protagonist: "从回避到直面"
  wife: "从旁观到参与"
open_questions:
  - "父亲信中提到的'故人'是谁"
  - "妻是否会看到遗物"
```

### 3.4 整故事摘要的字段（v0.1 草稿）

## 4. 每章极简摘要的格式

> 蕾姆酱的命名：把 §3.2 的"章末极简摘要"单独抽出来，**强调这是 LLM 必填的固定 schema**。

### 4.1 为什么强调"格式"

阶段摘要和整故事摘要**可以由 LLM 自由发挥**（让 LLM 觉得"我在写故事总结"）。

但章末极简摘要**必须是结构化**：
- **程序可解析**（用于种子表自动更新 / 阶段摘要自动生成）
- **不可省略字段**（key_events / seed_changes / unresolved 是必须填的）
- **长度硬上限**（150 token，超出截断）

### 4.2 Schema（v0.1 草稿）

```typescript
interface ChapterSummary {
  chapter_id: number;        // 章节编号
  range: string;             // "第 N 章（段 A-B）"
  key_events: string[];      // 1-3 句
  seed_changes: {
    planted?: number[];      // 本章新埋的种子 ID
    resonating?: number[];   // 本章强化的种子 ID
    harvested?: number[];    // 本章回收的种子 ID
  };
  character_state: Record<string, string>;  // { "protagonist": "疏离" }
  unresolved: string[];      // 存疑点
}
```

### 4.3 LLM 生成指令（v0.1 草稿）

放在章节生成 prompt 的**最后**（Assistant 输出之后作为下一个 System 指令）：

```markdown
请按以下 JSON 格式输出本章摘要，不要任何额外说明：

{
  "chapter_id": <当前章节号>,
  "range": "<第 N 章（段 A-B）>",
  "key_events": ["<事件 1>", "<事件 2>"],
  "seed_changes": {
    "planted": [<新种子 ID>],
    "resonating": [<强化种子 ID>],
    "harvested": [<回收种子 ID>]
  },
  "character_state": { "<角色名>": "<1-2 词状态>" },
  "unresolved": ["<存疑 1>", "<存疑 2>"]
}
```

**关键约束**：
- **强制 JSON**（不让 LLM 加解释性文字）
- **字段缺失 = 生成失败**（让程序校验）
- **新种子 ID 由程序分配**（不要让 LLM 写"种子内容"，种子内容是埋下时的 prose，不在摘要里）

### 4.4 种子表的自动更新

收到 ChapterSummary 后，程序自动：
1. `planted` 数组 → 在种子表加新行（base 由用户在 Step 3d 设定 / 或默认 0.5）
2. `resonating` 数组 → 把对应种子的 `last_seen_segment` 改为当前段
3. `harvested` 数组 → 把对应种子状态改为 `harvested`（不再进 prompt）

**这是程序责任，不是 LLM 责任**。
