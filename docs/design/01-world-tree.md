## 1. 世界树 (World Tree) · 核心隐喻与数据结构

### 1.1 树形结构

```
              ┌──────────────┐
              │   主线 · 树干  │  ← Step 3a 确认，结构层
              │  （核心矛盾）  │
              └──────┬───────┘
                     │
        ┌────────────┼────────────┐
        │            │            │
     ┌──┴──┐      ┌──┴──┐      ┌──┴──┐
     │支线A│      │支线B│      │支线C│  ← Step 3b 确认
     │(分枝)│      │(分枝)│      │(分枝)│
     └──┬──┘      └──┬──┘      └──┬──┘
        │            │            │
   ┌────┴────┐   ┌───┴────┐   ┌──┴────┐
   │ Node 1  │   │ Node 4 │   │ Node 7 │
   │ Node 2  │   │ Node 5 │   │ Node 8 │
   │ Node 3  │   │ Node 6 │   │ Node 9 │
   └─────────┘   └────────┘   └───────┘
   ← Step 5 剧情展开的"枝叶"，每章一个 Node

   ──── 树根（基座） ────
   · 题材共鸣（不变）
   · 风格锚点（极少变）
   · 核心人物关系（很少变）
```

### 1.2 生长机制

**故事不是"按时间累积"，是"按逻辑生长"**。

| 累积式（老思维） | 生长式（世界树思维） |
|---|---|
| 第 1 章 → 第 2 章 → 第 3 章…… | 主干 → 分枝 → 新芽 → 落叶 → 再分枝 |
| 时间是主线，剧情是子项 | 逻辑是主线，章节是表达 |
| 改设定要回头改前文 | 改基座 = 调整生长方向，**老枝叶保留作为"既成历史"** |
| 一致性靠"状态卡" | 一致性靠"树根基因"自洽 |

### 1.3 修改的代价层级（世界树表达）

| 修改类型 | 代价 | 世界树操作 | 用户权力 |
|---|---|---|---|
| 改某章某细节 | 极小 | 修剪一片叶子 | 演员级 |
| 改某章走向 | 小 | 让一根枝朝新方向长 | 演员+ |
| 改支线节点 | 中 | 嫁接/重剪一根枝 | 导演级 |
| 改主线走向 | 大 | 整根主干的某段重新长 | 编剧级 |
| 改核心人物关系 | 大 | 嫁接到新主枝 | 编剧级 |
| 改题材 / 基调 | 极大 | **移动树根** | 上帝级（极少用） |

**欧尼酱 20:54 确认**：
> 基座改了 → **老的枝叶保留**，新枝叶按新基座长出来
> （**B 解读**——不是修剪老枝叶，是"新枝叶按新基座长"）

→ **这意味着系统需要"存档回档"机制**（见 1.4）

### 1.4 存档与回档（单线树·硬 reset）

**欧尼酱 21:30 关键设计**（蕾姆酱此前的"分叉并存"假设被否定）：
> 当用户回档/返回指定 node 重新开始时，**历史基于该 node 生成的内容应该直接删除，只保留一棵干净的树**！（否则树枝无线扩散的管理成本太高）

**核心规则**：
- **世界树是单线树**——永远只保留一条主干
- **回档 = 硬 reset**——从回档点起，原先生成的所有枝叶（包括节点、种子、阶段摘要）**全部删除**
- 用户**不会看到"被放弃的内容"**——避免决策疲劳与历史污染
- **Node 仍然保留**（回档点之前的 Node 仍可回看）——只是**回档点之后的内容被裁掉**

**数据示意**（回档到 Node4 之后）：

```
              主线（未变）
                │
        ┌───────┼───────┐
        │       │       │
      支线A   支线B    支线C
        │       │       │
   Node1─2─3  Node4   Node7  ← Node5/6/8 被裁掉
                 │
                 ▼
             Node4'─5'─6'   ← 按新指令从 Node4 重新长出来
             （新种子注册表）
             （新阶段摘要）
```

**回档 vs 基座修改的明确区分**（重要）：

| 操作 | 本质 | 枝叶去留 | 数据状态 |
|---|---|---|---|
| **基座修改** | 调一下世界的"基因"，故事**继续** | 老的枝叶**保留**（B 解读）——但基座变了，老枝叶是"旧基座下的产品" | 同一棵树上加新约束 |
| **回档到某 Node** | 故事**重启**——这段不要了 | 回档点之后**全部删除** | 树被裁掉一段 + 新枝按新指令重长 |

→ **两个操作必须用不同的 UI 入口**——基座修改是"调整世界参数"，回档是"回到过去的某点"

**对世界契约的隐含要求**：
- Node 之前的章节永久保留（可回看）
- 回档点之后的内容**被裁掉后不可恢复**——这是**有意的设计**（避免用户陷入"分叉选择焦虑"）
- 种子注册表、阶段摘要、伏笔追踪——回档时**一并重置**
- 回档 ≠ 修改基座——是两个独立操作

### 1.5 种子与伏笔（同一概念）

**欧尼酱 21:30 明确**：
> 种子/伏笔是**一回事**，都是"计划要在后面某个节点实现的故事情节"

**统一术语**：
- **本文档统一用"种子"**——指"计划在后面某节点实现的故事情节"
- 包含但不限于：
  - **具体性颗粒的跨章复现**（v3.1 8.7）→ 是种子
  - **剧情伏笔**（"这个 NPC 是不是反派"）→ 是种子
  - **人物关系的变化**（"主角和师父会反目"）→ 是种子
  - **关键道具的下落**（"那把剑会回来"）→ 是种子
- **不区分"写作技法层"和"剧情层"**——都用同一套数据结构和加权算法

**结构化存储 + 加权算法**（v0.1 概要，详见 02 · 一致性方案）：

- 每颗种子记录：埋下时的 Node、计划复现的 Node 范围、当前权重
- **权重算法**示例：
  - 埋下后 5 段内 = 权重 1.0（必须复现）
  - 5-15 段 = 权重 0.5（建议复现）
  - 15+ 段 = 权重 0.2（弱提醒）
- **当权重到达阈值后**——将该种子写入 prompt
- **复现完成后**——标记"已落地"，权重清零

**对世界契约的隐含要求**：
- Step 3d 种子预埋**就是**种子的"初始注册"
- Step 4 产物 3 种子注册表**就是**种子的"结构化存储"
- Step 5 循环每章生成时**就是**种子的"权重计算 + prompt 注入"
- 种子库随回档**重置**——回档点之后的种子全部清空

## 2. 世界契约 · 启动链路（5 步）

### 2.0 链路总览

| Step | 名称 | 用户可见 | 关键产物 |
|---|---|---|---|
| **1a** | 必选标签 | ✅ | 题材 / 风格 / 基调 |
| **1b** | 可选标签（调色板） | ✅ | 言情 / 欢快 / 成人向 / 等 |
| **2** | 引导式自由文本 | ✅ | 多维度元素（带门禁） |
| **3** | 大纲确认 | ✅ | 3a 主线 / 3b 支线 / 3c 人物 / 3d 种子 |
| **4** | 后台准备 | ❌ 不可见 | 7 件产物 + 世界树框架 |
| **5** | 进入剧情 | ✅ | 节点 / 介入点 / 存档回档 |

### 2.1 Step 1a · 必选标签

**目的**：用最少的输入完成"基础筛选"

| 维度 | 类型 | 例子 | 是否必填 |
|---|---|---|---|
| **题材** | 多选 / 标签 | 修仙 / 都市 / 玄幻 / 古风 / 校园 | ✅ |
| **风格** | 多选 / 标签 | 战斗 / 生存 / 悬疑 / 治愈 | ✅ |
| **基调** | 单选 / 滑块 | 压抑 / 温暖 / 残酷 / 治愈 | ✅ |

**交互**：
- 至少选 1 个题材 + 1 个风格 + 1 个基调
- **不阻塞后续**——选得少也只是"低锚定"标记，不是错误

### 2.2 Step 1b · 可选标签（调色板）

**目的**：让用户用"加减法"调整偏好，不强制

**例子**：言情、欢快、成人向、群像、单女主、慢热、快节奏……

**交互**：
- 用户可跳过
- 可随时增减
- **这一类标签不参与"门禁"**——只是"装饰性偏好"

### 2.3 Step 2 · 引导式自由文本（带门禁）

**目的**：通过**多段对话引导**，采集用户的"明确多维度元素"

**关键设计**：
- **不是一次性表单**，是**多轮对话**（3-5 轮）
- **每轮聚焦一个维度**（"你希望主角是个什么样的人？""你希望看到什么样的关系？""你不想看到什么？"）
- **回环机制**：用户说完后，系统**复述 + 确认**（"我理解你想看的是：xxx 对吗？"）

**门禁设计（v0.1 草案，待精化）**：

| 类别 | 元素 | 必填程度 |
|---|---|---|
| 题材 | 修仙 / 都市 / 玄幻 / ... | ✅ 硬必填（已在 Step 1a 选过） |
| 基调 | 压抑 / 温暖 / 残酷 / 治愈 | ✅ 硬必填（已在 Step 1a 选过） |
| 核心关系 | 师徒 / 敌对 / 暧昧 / 兄妹 / ... | ✅ 硬必填 |
| 情感锚点 | "想看被压抑很久的情感爆发" 这类自由文本 | ⚠️ 软必填（≥1 即可） |
| 禁区 | "不要后宫 / 不要种马 / 不要系统流" 这类反例 | ⚠️ 软必填（≥1 即可） |
| 结局倾向 | 必好 / 必悲 / 开放 | ❌ 可选 |

**门禁规则**：
- **硬必填 ≥ 3 项** → 进入 Step 3
- **软必填 < 1 项** → 提示用户"再聊聊 XXX"（不阻塞但提示）
- **全部空白** → 进入 Step 3，但标记"低锚定"风险（系统不乱猜，而是让 Step 3 用结构化问题补）

**欧尼酱 20:54 关键洞见**：
> 元素丰富度取决于大纲的内容丰富度
> （简单大纲 = 3-4 个元素；复杂大纲 = 5-7 个元素；史诗 = 7+）

→ **这是 Step 2 ↔ Step 3 的反馈环**——

```
Step 2 元素采集
      ↓
Step 3 用户选定大纲复杂度
      ↓
系统判断 Step 2 元素是否足够
      ↓
   不足 → 返回 Step 2 提示补充
   足够 → 进入 Step 4 后台准备
```

**待设计项**（不写进 v0.1）：
- 反馈环的精确化（什么算"不足"）
- 软必填的"再聊聊"具体怎么问
- 低锚定故事的实际效果（不乱猜的边界）

### 2.4 Step 3 · 大纲确认

**目的**：用户和系统**共同敲定**世界树的"框架"

**介入程度滑块**（用户开场时选）：

| 模式 | 用户参与度 | 适用 |
|---|---|---|
| **完全自由发挥** | 极低 | 用户开完设定就放手 |
| **中介入** | 中 | 开场大纲确认，章内自由 |
| **深度讨论** | 高 | 每章大纲都跟用户确认 |

**深度讨论模式下的 4 类结构化问题**：

#### 3a. 主线预设（必做）
- **故事核心矛盾**（1 句话）
- **主线 5-7 个关键节点**（约 50% 是"必须发生"，50% 是"可能发生"）
- **结局倾向**（必好 / 必悲 / 开放）

#### 3b. 支线设计（可选）
- **主要支线 1-2 条**
- 每条支线的关键节点
- 支线与主线的交叉点

#### 3c. 人物关系图（必做）
- **主要人物 3-5 个**（含主角）
- **关系矩阵**（师徒 / 敌对 / 暧昧 / 兄妹 / 等）

#### 3d. 种子预埋（必做）
- **2-3 颗"具体性种子"**（v3.1 8.7 原则）
- 每颗种子的"原型"和"计划复现场景"
- **这步直接落地 v3.1 的种子延续原则**

**用户回答方式**：
- 高介入模式：每条都跟用户**逐项确认**
- 中介入模式：系统**生成草案**，用户**修改关键点**
- 低介入模式：系统**自己定**，用户**只看不改**

### 2.5 Step 4 · 后台准备（**7 件产物**）

> **2026-06-12 升级记录**：产物数量从原 6 件改为 7 件（详见 02-consistency.md §5）。
> 旧 6 件（世界观设定卡 / 人物关系树 / 种子注册表 / 密度配比表 / 风格宪法 / 大纲骨架）
> → 新 7 件（世界树 / 风格宪法 / 题材共鸣 / 主线大纲 / 支线大纲 / 人物卡 / 种子表）
> 映射表见本文末尾「附录 · 旧 6 件 → 新 7 件 映射表」。

**⚠️ 这一步**一定一定不能让用户看到**——否则沉浸式效果大打折扣**（欧尼酱原话）

**7 件产物**：

| # | 产物 | 用途 | 与 v3.1 衔接 | 主要消费者 |
|---|---|---|---|---|
| 1 | **世界树** | 时间线 / 地理 / 核心规则 | — | 基座层 |
| 2 | **风格宪法** | 4 维原则（v3.1 公式） | ✓ | 基座层 |
| 3 | **题材共鸣** | 用户接受什么 + 拒绝什么 | ✓ | 基座层 |
| 4 | **主线大纲** | 主线节拍 | — | 动态调参层 |
| 5 | **支线大纲** | 支线故事 | — | 动态调参层 |
| 6 | **人物卡** | 角色 + 关系 + 弧光 | — | 动态调参层 |
| 7 | **种子表** | 跨章复现管理（v3.1 8.7），4 维权重 | ✓ | 动态调参层 |

**这 7 件产物 = 世界树的"基因 + 结构"**——生成每一章时都被读取。

### 2.6 Step 5 · 进入剧情

**循环**：
```
[每章循环]
  │
  ├─→ 读取世界树 + 7 件产物
  │
  ├─→ 生成 3000+ 字的章节
  │     · 颗粒数/千字 ≤ 3（v3.1 8.6）
  │     · 种子复现 ≥ 1 次（v3.1 8.7）
  │     · 密度配比动态微调（v3.1 8.4）
  │     · LLM-as-judge 自查
  │
  ├─→ 输出介入点（导演/演员模式）
  │
  └─→ 存档为 Node
       · 可被回看
       · 可作为回档点
```

**用户可触发的操作**：
- **继续生成**（按种子 + 大纲 + 当前介入意图）
- **回看 Node**（重读已生成的章节）
- **回档重新出发**（从某 Node 起，按新指令生长）
- **修改基座**（极少用——上帝级，**老枝叶保留**，新枝按新基座长）

## 4. 与 v3.1 的衔接（沉浸感公式如何落进世界契约）

| v3.1 公式 | 世界契约里的"宿主" |
|---|---|
| **具体性** | Step 5 生成的硬约束（颗粒数/千字 ≤ 3） + 种子复现驱动 |
| **主观性** | Step 4 产物 2（人物关系树） + 5（风格宪法） |
| **密度** | Step 4 产物 4（密度配比表） + 5（风格宪法），静态 + 动态微调 |
| **题材共鸣** | Step 1a 必选 + Step 2 引导式自由文本 |
| **种子跨章复现** | Step 3d 种子预埋 + Step 4 产物 3（种子注册表） + Step 5 加权算法 + prompt 注入 |
