## 0. 文档目标

**这一份要回答的问题**：怎么**测**"一致性 / 沉浸感 / 种子加权"三个核心机制是否有效？

**为什么必须有这份文档**：
- 01 立项、02 一致性、03 schema 都讲"系统**应该**怎么工作"
- 04 讲"系统**实际**怎么工作"——用数据证明设计是对的
- **没有评测 = 没有产品**：再漂亮的设计，无法验证就等于猜测

**不回答的问题**：
- 真实用户反馈怎么收（这是 PM 范畴，不在 v0.1 评测里）
- 怎么用真人评估（v0.1 全部自动化评测，真人评估留 v0.2）

**核心思路**：

```yaml
评测目标: 用可量化的、自动化的、可重复的指标
         验证 01/02/03 设计在 3 个核心机制上是否达到预期
3 个核心机制:
  - 一致性: 长程记忆是否保持（种子回收 / 角色不 OOC / 基座不违反）
  - 沉浸感: 00 v3.1 模型是否被准确实现（具体性 / 主观性 / 密度 / 题材共鸣）
  - 种子加权: 02 §2 的 4 维权重算法是否按预期影响 LLM 行为
```

---

## 1. 评测原则

```yaml
评测原则:
  离线: 不依赖真实用户（v0.1 全部跑自动化评测）
  可重复: 同一输入 + 同一模型 = 同一输出（用 LLM 温度 0 / seed 固定）
  可量化: 指标是数字（百分比 / 比率 / 计数），不是感觉
  分层: 三层指标对应三个核心机制（不混）
  边界明确: 什么测得到、什么测不到，文档里写清楚
  LLM-as-judge: 涉及"风格 / 主题"的指标用 LLM 评判（参考 00 v3.1 8.6）
```

**为什么用 LLM-as-judge**：
- "角色说话风格是否一致"——这是"感觉"，无法用关键词匹配
- 00 v3.1 8.6 已确认 LLM-as-judge 是 v3.1 沉浸感自评的合法机制
- **风险**：LLM-as-judge 自身有偏差（不同 LLM 给的分数不同）——v0.1 用同模型 judge 自己（少偏差但可能 self-confirm），v0.2 再考虑 cross-judge

## 2. 三类指标

### 2.1 一致性指标

```yaml
一致性指标:
  种子回收率:
    定义: 埋下的种子在 N 章内被再次提及的比例
    计算: 回收数 / 埋下数
    目标: '>= 70%'
    衡量: 02 §2 4 维权重算法是否有效
    测不了什么: 回收得好不好 - 需要 LLM-as-judge 辅助
    
  角色行为一致率:
    定义: 角色在 N 章后的 speech_style / traits / arc 与首次出现时一致的比例
    计算: 一致章数 / 总章数
    目标: '>= 85%'
    衡量: CharacterCard 的人物设定是否被 LLM 遵守
    测不了什么: 微妙的 OOC - 需要 LLM-as-judge
    
  基座约束遵守率:
    定义: 违反 WorldTree.core_rules[enforcement=hard] 的次数
    计算: 违反次数 / 总章数
    目标: 0%
    衡量: 基座层在 prompt 中的约束力
    测不了什么: soft 规则的软违反
    
  题材共鸣匹配度:
    定义: 用户 reject 列表中的元素在生成章节中出现的比例
    计算: 1 - 出现次数 / 总章数
    目标: '>= 95%'
    衡量: GenreResonance 约束力
    测不了什么: 隐性出现 - 需要 LLM-as-judge
```

**蕾姆的判断**：一致性指标是**最关键**的——产品失败的最大风险是"前后矛盾"，4 个指标覆盖 4 个一致性维度。

---

### 2.2 沉浸感指标（继承 00 v3.1）

```yaml
沉浸感指标:
  具体性颗粒密度:
    定义: 每千字的具体性颗粒数
    计算: （具体词 / 数字 / 专有名词 总数）/ 字数 × 1000
    目标: ≤ 3 / 千字（v3.1 硬约束）
    衡量: v3.1 8.6 硬上限
    检测: 关键词字典 + LLM-as-judge 复核
    
  主观性词占比:
    定义: 主观性描述（"我觉得" / "他似乎" / 心理活动）占总词数的比例
    计算: 主观词数 / 总词数
    目标: 0.05-0.15（区间，下限防止无心理活动，上限防止汉尼拔式过载）
    衡量: v3.1 8.4 密度配比
    检测: 关键词字典（"觉得"/"似乎"/"仿佛"/"大概" 等 30+ 词）
    
  密度分:
    定义: 段落的"信息密度"——具体词 + 主观词 + 数字 / 段落字数
    计算: 段落级 → 章节级 → 加权平均
    目标: 0.3-0.6（区间）
    衡量: v3.1 8.4
    检测: 综合上面两个指标
    
  题材共鸣准确率:
    定义: 用户 accept[] 中的元素在生成章节中"被用对"的次数
    计算: 准确使用次数 / 接受元素总次数
    目标: ≥ 80%
    衡量: GenreResonance.accept 的命中质量
    检测: LLM-as-judge（"这段是不是'日常细节'，还是'巧合煽情'"）
```

**蕾姆的判断**：沉浸感指标**部分自动化**（具体性 / 主观性 / 密度）+ **部分 LLM-as-judge**（题材共鸣准确率）。**纯字典匹配会漏掉语义层面**。

---

### 2.3 种子加权指标

```yaml
种子加权指标:
  overdue 触发命中率:
    定义: overdue_score ≥ 0.7 的种子在下一章 prompt 中出现的比例
    计算: 命中数 / overdue 数
    目标: ≥ 90%
    衡量: 02 §2.3 双重缩放 + sigmoid 是否按预期
    检测: 抓取 prompt 中的种子 ID，与 SeedTable.weight 对比
    
  importance 优先采纳率:
    定义: importance=主线推进 的种子被 LLM 选用的比例
    计算: 选用数 / 主线推进种子数
    目标: ≥ 80%
    衡量: 02 §2.4 enum 预设数值表是否合理
    检测: 对比 prompt 中的种子 importance 分布 vs LLM 实际选用
    
  planned_interval 到期及时性:
    定义: 种子的 planned_interval 到期后，在 ±3 章内被回收的比例
    计算: 及时数 / 总到期数
    目标: ≥ 75%
    衡量: 02 §2.6 种子导向是否驱动 LLM 按时回收
    检测: 跟踪种子的 harvested 状态与时间
    
  weight 排序影响:
    定义: prompt 中种子出现顺序与 weight 排序的一致性
    计算: （一致位置数）/（总位置数）
    目标: ≥ 85%
    衡量: 02 §2.2 权重公式 + §2.9 prompt 注入格式是否生效
    检测: token 截断是否按 weight 降序
```

**蕾姆的判断**：种子加权是**最复杂的评测**——3 个独立维度的设计需要分别验证。**这套指标不达标，02 文档的整套机制要回炉**。

## 3. 评测用例集

**为什么需要固定用例集**：没有固定输入，所有指标都是"今天跑出来这样"——不可对比、不可调优。

```yaml
用例集设计原则:
  数量: 5 个用例（v0.1 起步，少而精）
  覆盖: 3 种题材（都市 / 玄幻 / 科幻）+ 2 种长度（短篇 20 章 / 长篇 80 章）
  预期产出: 每个用例有"预期种子列表 + 预期一致性检查点 + 预期沉浸感区间"
  形式: JSON 输入（7 件产物的初始化数据）+ YAML 预期（指标目标值）
```

**用例 1 草案**（开个头，剩下 4 个同结构）：

```yaml
用例 1 - 都市情感:
  题材: 现代 / 杭州 / 程序员
  长度: 20 章
  核心人物: 主角 28 岁程序员 + 妻（高中语文老师）+ 父亲（已故，遗物）
  
  预期种子:
    - id: 1
      content: 1987 年的收音机
      importance: 主线推进
      size: 中线
      planned_interval: 10 段
      orientation: 主角成长
      预期回收章: 第 8-12 章
    - id: 2
      content: 父亲的信
      importance: 主线推进
      size: 长线
      planned_interval: 15 段
      orientation: 关键成员关系
      预期回收章: 第 15-18 章
    - id: 3
      content: 街角早餐店
      importance: 支线故事
      size: 点状
      planned_interval: 5 段
      orientation: 氛围营造
      预期出现章: 第 3 章
  
  预期一致性检查点:
    - 第 5 章: 主角 28 岁，妻是老师（基座遵守）
    - 第 10 章: 收音机再次出现（种子 1 回收）
    - 第 15 章: 信被读出（种子 2 回收）
  
  预期沉浸感区间:
    具体性颗粒: 2.0-2.8 / 千字
    主观性占比: 0.08-0.12
    题材共鸣匹配: ≥ 0.85
```

**蕾姆酱的执行计划**：用例集不全在这一份文档里写——**v0.1 评测阶段再展开 5 个完整用例**，这里只给 1 个示例定调。

## 4. 评测 Pipeline

```yaml
pipeline 总览:
  阶段一 - 单用例离线生成:
    输入: 7 件产物 JSON
    过程: 按 02 §1 三层 prompt 结构生成 N 章
    输出: 生成的章节 + 章节摘要 JSON（按 02 §4.3）
    工具: 调 friday 代理（MiniMax-M3）+ 温度 0 + seed 固定
  
  阶段二 - 指标计算:
    输入: 阶段一输出
    过程:
      - 一致性指标（自动检测为主 + LLM-as-judge 为辅）
      - 沉浸感指标（自动检测）
      - 种子加权指标（自动检测）
    输出: 指标 JSON（每个用例一份）
  
  阶段三 - A/B 对照:
    输入: 多组"相同输入 + 不同算法变体"的阶段一输出
    过程: 对比指标差异
    输出: A/B 对照报告
    用法: 调优 02 §2 的 k 值 / overdue_score 下限 / 阈值 0.7/0.3
  
  阶段四 - 边界用例:
    输入: 特殊场景（回档 / 基座修改 / 种子过期）
    过程: 验证系统在边界场景下的指标
    输出: 边界场景报告
    用法: 验证 03 §1/§2 的回档行为是否正确
```

**执行节奏**（蕾姆建议）：
- 阶段一/二 跑 5 个用例 × 20 章 = 100 章生成，每次约 30-60 分钟
- **不必全跑**——v0.1 先跑用例 1 单点验证评测机制本身能跑通，再扩展

---

## 5. 评测的边界（v0.1 测不到什么）

```yaml
v0.1 测不到:
  - "故事好不好看": 这是主观审美，v0.2 用真人 A/B 测
  - "用户会不会继续读": 这是产品指标，v0.2 接真实用户数据
  - "LLM-as-judge 的偏差": v0.1 接受 self-judge 偏差，v0.2 引入 cross-judge
  - "种子加权 vs 不加权的差异": v0.1 只测有加权时的指标，v0.2 加对照组
  - "回档后用户满意度": 这是 UX 范畴，v0.2 调研
  - "300+ 章远期表现": v0.1 用例最长 80 章，v0.2 推 200+ 章

v0.1 能测:
  - 3 个核心机制的设计**是否符合预期**（指标达成度）
  - 03 schema 的字段**是否被实际使用**（空字段检测）
  - 02 三层 prompt 结构**是否被 LLM 理解**（prompt 注入格式检查）
  - 01 启动链路**是否产生预期的 7 件产物**（初始化完整性）
```

**蕾姆的诚实判断**：04 v0.1 **只能测"系统按设计工作"**，**不能测"设计是否对"**。后者要等 v0.2 用真实用户反馈。

### 6.2 蕾姆酱拍板的 3 个决策（不拖欧尼酱）

| 决策点 | 蕾姆的拍板 | 理由 |
|--------|-----------|------|
| **v0.1 跑几个指标** | **5 个核心**：种子回收率 / 基座约束遵守率 / 具体性颗粒 / overdue 触发命中率 / importance 优先采纳率 | 5 个分别覆盖 3 大类，避免 v0.1 一上来跑 12 个 |

### 6.3 蕾姆酱学到的事

- **评测是闭环**：01→02→03 是"设计"路径，04 是"验证"路径，**两者合起来才叫产品**
- **可量化 vs 可执行**：指标设计既要数学上可量化，也要工程上可执行（关键词字典 + LLM-as-judge 混合）
- **边界明确**比"什么都测"重要——04 §5 写"v0.1 测不到什么"比"v0.1 能测什么"更体现产品负责人诚实
- **v0.1 用例集不全写**是正确决策——04 是"评测方法"不是"评测数据"，用例集是"评测实例"
- **扛责的延续**：欧尼酱 16:38 那句"不要让我拍细节"在 04 继续生效——12 个指标蕾姆自己选了 5 个 v0.1 跑
