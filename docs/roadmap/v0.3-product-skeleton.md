## 1. 方向调整对照

| 维度 | 调整前（05 主线）| 调整后（本文件主线）|
|------|------------------|---------------------|
| **首要目标** | M1 算法调优 → 5 指标真实 baseline | **全链路骨架搭建** → 产品可运行 |
| **代码产出** | `eval/v0.3/` 评测脚本 | **`realtime_novel/` 产品骨架包** |
| **度量标准** | 5 指标达标率 | **骨架可加载 + 端到端跑通** |
| **算法** | 立刻调 `planned_interval` | **M-α 阶段不调**，原值沿用 |
| **并行关系** | M1-M4 串行 | **骨架与算法并行**：骨架先成型，调优在骨架上做 |

**关键原则**：
1. **骨架优先** —— 没有可用的产品代码，再准的算法调优也只是评测玩具
2. **算法沿用** —— v0.2 的 4 维权重、planned_interval、sigmoid k 值全部沿用
3. **接口先行** —— S1-S5 先定接口，算法层换实现不影响产品侧
4. **可回滚** —— 每个骨架版本的产物 + 代码都落盘，不破坏 v0.2 评测
5. **配置集中** —— 复用 v0.2 LLM 配置（`~/AiTest/lunaris/config.private.json`）

## 3. 骨架 5 件套（S1-S5）

### S1 · 项目目录管理器

**职责**：创建 / 加载 / 列出项目，装载 7 件 YAML 产物。

**目录结构**：
```
projects/{project_id}/
├── manifest.yaml              # 0 · 项目元信息
├── genre.yaml                 # 1 · 题材
├── foundation.yaml            # 2 · 基座
├── main_plot.yaml             # 3 · 主线
├── branches.yaml              # 4 · 支线
├── seeds.yaml                 # 5 · 种子库
├── summaries.yaml             # 6 · 摘要链
├── current_node.yaml          # 7 · 当前 Node + 状态
└── chapters/
    ├── chapter-01.txt
    └── ...
```

**依赖**：`design/03` 的 7 件 Schema

### S2 · WorldTree 数据结构

**职责**：Node / 主线 / 支线 / 种子 / 摘要 的内存模型 + YAML 序列化。

**关键设计点**（来自 `design/01 §1.4`）：
- **单线树** —— 永远只保留一条主干
- **回档硬 reset** —— 从回档点起，原先生成的所有枝叶全部删除
- **Node 保留** —— 回档点之前的 Node 仍可回看

**依赖**：`design/01 §1` 树形结构 + `design/03` 各 YAML 格式

### S3 · 启动链路 orchestrator

**职责**：5 步引导用户生成 7 件产物。

**实现方式**：M-γ 阶段用 CLI 实现（最快验证骨架），M4 阶段再考虑 Streamlit / Textual / Web UI。

**依赖**：`design/01 §2` 启动链路设计 + S1 + S2

### S4 · 章节生成 orchestrator

**职责**：输入干预 → 组装 prompt → 调 LLM → 写章节 → 更新 WorldTree。

**关键设计点**：
- **3 层 prompt** —— 来自 `design/01-03` 各处
- **种子提取** —— 生成后从章节文本提取新种子
- **摘要滚动** —— 每 15 章生成一次阶段摘要
- **字数校验** —— `design/00` 要求 ≥ 3000 字

**依赖**：S2 + L3 LLM 客户端（复用 eval v0.2）

### S5 · 干预 + 回档 orchestrator

**职责**：导演/演员输入解析 + 回档硬 reset。

**关键设计点**（来自 `design/00 §2.2` + `design/01 §1.4`）：
- **双视角双干预语义** —— 第三视角是引导、第一视角是入戏
- **回档不破坏历史 Node** —— Node 仍可回看
- **保险箱 `.trash/`** —— M-α 简化版先不做

**依赖**：S2 + L1 持久化（文件清理）

- [ ] 生成 1 章真实产品流

## 5. 远期参考（M1-M4 · 原 05 路线图）

> 以下来自原 `05-roadmap-v03.md`，收为"远期参考"——骨架搭完再回头看。

### M1 · v0.3 真实长程评测

- 跑 20 章真 LLM（已在 v0.2 完成）
- 跑 80 章真 LLM（v0.3 新增）
- 验证 5 个核心指标在长程下的真实表现
- 5 指标的真实 baseline（用于 M2 调优）

### M2 · 02 §2 算法调优（基于 M1 数据）

- 如果 5 指标未达标，调优 `design/02 §2` 的 4 维权重算法
- 如果 overdue 命中率低，调 sigmoid k 值
- 如果 importance 优先采纳率低，调预设数值表
- A/B 对照报告（v0.2 vs v0.3 调优后）

### M3 · 远期架构（300+ 章挑战）

- 解决"故事不设上限，300+ 章"的压缩挑战
- 阶段摘要从 15 章固定 → 动态触发
- 整故事摘要从 3 份 → 几何滚动窗口（30/60/120/240/480）
- 引入向量检索（Mnemosyne 现成可用）做"远期记忆"

### M4 · 用户 UI + 多 agent

- 启动链路 5 步的用户界面设计
- 每次章节生成后的用户介入 UI（导演/演员模式）
- 多视角（双视角导演/演员）的 agent 协作

### 远期总估时

| 里程碑 | 预计耗时 | 关键风险 | 缓解方案 |
|--------|---------|---------|---------|
| **M1 v0.3** | 1-2 天 | 20/80 章真 LLM 跑通 + 5 指标真实 baseline | 后台长跑 + 增量报告 |

## 6. 验收标准 & 风险登记

### 6.1 总验收

骨架 0.1（M-α）跑通 = **能用 `python3 -c "..."` 加载 demo + 操作 WorldTree**。

### 6.2 风险登记

| 风险 | 等级 | 缓解方案 |
|------|------|---------|
| S2 数据结构与 v0.2 mock 数据不兼容 | 中 | 优先对齐 design/03 schema，启动时校验失败即报错 |
| LLM 配置路径错误 | 低 | 沿用 v0.2 已知路径，启动时打印确认 |
| 算法沿用导致 80 章 overdue 仍不触发 | 低 | 不是本里程碑目标，5-M1 范围 |
| WorldTree 序列化 YAML 体积爆炸 | 中 | 分文件存储（7 件产物），不存整树 |
| 回档实现 bug 污染 WorldTree | 高 | M-δ 阶段必须加单元测试覆盖回档场景 |
| CLI 交互体验差 | 低 | M4 阶段优化，本里程碑目标"能用"即可 |

### 6.3 回滚策略

- 每个骨架版本独立目录（不破坏前一版）
- 不修改 `docs/eval-notes/code/` 下任何文件
- 不删除 `generated-stories/` 任何章节
- 7 件 YAML 变更必须有迁移脚本（如有）
- 重大结构调整前先 `git tag v0.X-final` 留保险点
