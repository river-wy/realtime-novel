## 0. 为什么不是「CLI 转 HTTP」那么简单

### 0.1 旧范式（v0.3 已做的）

```
用户 → 页面按钮 → 后端 CLI/HTTP 命令 → 调 LLM → 返回结果
```

**本质**：用户和**脚本**对话，LLM 只是脚本里的一个函数。

**问题**：
- 每个动作需要单独的按钮（生成下一章、回档、干预、调整基座、查项目…）
- 按钮之间没有"上下文"——用户每次都要重新告诉系统要做什么
- 复杂的「我想让主角遇到一个赛博朋克的反派，但保留他原来的性格」这种请求，**脚本根本处理不了**
- 用户在 onboarding 5 步里说「我喜欢悬疑」，到阅读时系统已经忘了这个偏好

### 0.2 新范式（v0.4 要做的）

```
用户 → 页面 → 对话 Agent「小说家」（主入口）→ 决定要做什么
                              ↓ 调度
        ┌─────────────────────┼─────────────────────┐
        ↓                     ↓                     ↓
   章节生成 Agent         世界树维护 Agent      工具能力
   （专门写章节）         （维护 7 件 + node 树）  （查/改/落盘）
```

**本质**：用户和**一个真实的存在（Agent）**对话，Agent 自己决定调谁、调什么工具。

**优势**：
- **自然语言入口**：「让主角遇到一个赛博风格的对手」→ Agent 自己拆解成「回档到第 15 章 + 改基座 + 重新生成」
- **跨动作上下文**：用户说「我刚才说的悬疑」→ Agent 能理解
- **可扩展**：未来加新能力（生成插图、推荐相似小说、多人协作），加 Agent 就行

**关键设计点**：
- **主 Agent「小说家」** 是入口（所有用户对话先到这里）
- **专家 Agent** 分两类：
  - **简单任务**（世界树/记忆/立绘）：直接函数调用即可（不需要多 Agent 协作）
  - **复杂任务**（章节生成）：W3 后用 AutoGen 实现多 Agent 协作
- **工具层**是 Agent 的"手"——所有副作用（写文件、调 LLM、落库）都封装成工具
- **持久化**双轨：文件保留（项目+章节）+ 新 SQLite（对话+日志+Agent 状态）

## 3. 对话状态机（用户消息 → Agent 调度）

### 3.1 主 Agent 决策循环（LangGraph）

```
用户消息进来
    ↓
[思考节点] LLM 分析: 用户想要什么？
    ↓ 输出: intent (generate / intervene / rollback / adjust_base / create_project / chat)
    ↓
[路由节点] (条件边) 根据 intent 分发
    ↓
    ├── generate        → 章节生成 Agent → 等结果 → 流式返回用户
    ├── intervene       → 章节生成 Agent (带干预 prompt) → 等结果 → 流式返回
    ├── rollback        → 工具: rollback_to() → 工具结果回流 → 返回
    ├── adjust_base     → 世界树维护 Agent → 改 7 件 → 工具结果回流 → 返回
    ├── create_project  → 工具: create_project() + 启动 OnboardingFlow → 返回
    ├── chat (闲聊)     → LLM 自由回答 (基于对话历史) → 返回
    └── unknown         → LLM 自由回答 + 建议用户怎么问
    ↓
[工具结果回流] 工具执行结果回主 Agent
    ↓
[回复生成节点] LLM 把结果组织成自然语言回复
    ↓
流式推送给前端（WebSocket）
```

### 3.2 章节生成 Agent 内部（AutoGen W3+）

```
主 Agent: "用户想生成下一章"
    ↓
章节生成 Agent (AutoGen GroupChat)
    ├─ "剧情分析师" Agent: 找当前 beat，决定情节走向
    ├─ "风格师" Agent: 调取项目调色板 + 风格约束
    ├─ "写手" Agent: 综合上面，输出章节正文
    └─ "质检" Agent: 校验字数 + 角色一致性 + 钩子
    ↓
生成章节文件 → 落盘 → 更新 WorldTree
    ↓
结果回流主 Agent
```

**W1 简化版**（先出 MVP）：用 LangGraph 单 Agent 完成，不引 AutoGen。
**W3+ 升级版**：换成 AutoGen 多 Agent 协作。

## 5. 6 个 API 路由（5 资源 + 1 对话）

| 路由 | 方法 | 作用 | 后端调用 |
|------|------|------|----------|
| `/api/chat` | WebSocket | **对话接口**（流式推送 Agent 思考过程）| 主 Agent「小说家」 |

| `/api/projects/{id}/chapters` | GET | 列章节 | 读文件系统 |
| `/api/projects/{id}/chapters/{n}` | GET | 读章节正文 | 读文件系统 |
| `/api/projects/{id}/chapters` | POST | 生成下一章 | `generate_chapter` 工具 |
| `/api/projects/{id}/onboarding` | POST | 5 步启动链路 | OnboardingFlow 现有 |

| `/api/info` | GET | 版本信息 | — |

**对话接口是核心** — 旧的 5 资源接口保留（前端可以继续按按钮触发），但所有按钮**最终也走对话 Agent 转发**到工具。

```
│   │   ├── checkpoint_data (json)
│   │   └── updated_at
│   └── user_preferences # 用户偏好
│       ├── user_id
│       ├── key
│       ├── value
│       └── updated_at
```

**关键**：
- LangGraph 自带 SQLite checkpointer，存主 Agent 状态（`agent_state` 表）
- 对话历史（`conversations` + `messages`）支撑「记住之前说的」能力
- 工具调用日志（`tool_calls_log`）做审计和调试

## 7. 风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| AutoGen v0.4 API 不稳定 | 🟠 高 | 锁版本 + 写 adapter 层（未来换实现不改业务）|
| LangGraph 状态机表达力不够 | 🟢 低 | 单 Agent 场景已验证过（v0.2 用 LangChain 写过简单 ReAct）|
| WebSocket 复杂度 | 🟡 中 | SSE 备选（Server-Sent Events，单向流式）|
| SQLite 性能瓶颈 | 🟢 低 | 单机 1000 用户以下，SQLite 够用 |
| LLM 思考延迟让用户等待 | 🟡 中 | 流式推送中间态（"正在查项目..." "正在生成..."）|
| Agent 调错工具（如删错项目）| 🟠 高 | 危险操作必须 confirm=True，前端二次确认弹窗 |
| 对话历史无限增长 | 🟢 低 | 自动摘要压缩（旧消息 → 摘要，保留最近 N 条）|
| 多 Agent 调试困难 | 🟠 高 | LangGraph Studio + 自研日志中间件 + 工具调用审计 |
