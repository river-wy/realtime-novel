## 1. 里程碑（M1 - M4）

### M1 · v0.3 真实长程评测（最优先）

```yaml
M1 目标:
  - 跑 20 章真 LLM（已在 v0.2 启动）
  - 跑 80 章真 LLM（v0.3 新增）
  - 验证 5 个核心指标在长程下的真实表现

M1 关键决策:
  - 20 章章节是否触发 overdue 种子回收
  - 具体性颗粒密度在 20+ 章是否稳定 ≤ 3/千字
  - importance=主线推进 的种子是否被 LLM 优先采纳

M1 输出:
  - v0.2 20 章报告（v0.2 输出）
  - v0.3 80 章报告
  - 5 指标的真实 baseline（用于 v0.4 调优）
```

### M2 · 02 §2 算法调优（基于 M1 数据）

```yaml
M2 目标:
  - 如果 5 指标未达标，调优 02 §2 的 4 维权重算法
  - 如果 overdue 命中率低，调 sigmoid k 值
  - 如果 importance 优先采纳率低，调预设数值表

M2 关键决策:
  - k=2 是否合理（要改成 1 还是 3？）
  - overdue_score 下限 0.5 是否合适
  - importance/size/orientation 三档还是更多档

M2 输出:
  - 调优后的 02 §2 公式
  - A/B 对照报告（v0.2 vs v0.3 调优后）
```

### M3 · 远期架构（300+ 章挑战）

```yaml
M3 目标:
  - 解决 02 §3 "故事不设上限，300+ 章"的压缩挑战
  - 阶段摘要从 15 章固定 → 动态触发
  - 整故事摘要从 3 份 → 几何滚动窗口（30/60/120/240/480）
  - 引入向量检索（Mnemosyne 现成可用）做"远期记忆"

M3 关键决策:
  - 向量检索 vs 压缩摘要 的成本/质量权衡
  - 阶段长度动态计算公式（按密度/剧情张力）
  - "哪些种子值得永久记住"的筛选标准

M3 输出:
  - 02 §3 重设计（v0.4 文档）
  - 向量检索层设计
```

### M4 · 用户 UI + 多 agent

```yaml
M4 目标:
  - 01 启动链路 5 步的用户界面设计
  - 每次章节生成后的用户介入 UI（导演/演员模式）
  - 多视角（双视角导演/演员）的 agent 协作

M4 关键决策:
  - 启动链路是 Web UI / CLI / IM ？
  - 双视角协作时"世界树的枝"如何表达
  - 真人介入频率（每章 1 次 / 3 章 1 次 / 自由）

M4 输出:
  - UI 草图 + 交互流程
  - 多 agent 协作设计文档（v0.5 文档）
```

## 4. 关键设计原则（蕾姆扛责版）

1. **先验证再扩展**（v0.2 跑通后才做 v0.3 80 章）
2. **小步迭代**（一次只调一个参数，看指标变化）
3. **数据说话**（所有调优都基于真实 LLM 输出，不靠猜测）
4. **可回滚**（每个版本的产物 + 代码都落盘，不破坏前一版）
5. **配置集中**（模型、temperature、max_tokens 走环境变量，不硬编码）
