# realtime-novel v0.5 API 文档（前端对接专用）

> v0.5 相对 v0.4 的变化 + 14 个端点的人话版 + 前端需要的 3 个新增能力  
> 自动 OpenAPI 文档：`/docs`（Swagger UI）/ `/openapi.json`（JSON schema）  
> ReDoc：`/redoc`

**端口约定（v0.6 起）**：
- 后端 API：`http://127.0.0.1:7778`（启动：`uvicorn realtime_novel.api.app:app --port 7777`）
- 前端 dev：`http://localhost:7777`（启动：`cd frontend && npm run dev`）
- Vite 代理：`/api` 自动转发到 7777（前端代码只用 `/api`，不要写死端口）
- v0.4 文档里的 8080 是历史端口，请忽略。

## v0.5 相对 v0.4 的关键变化（前端必读）

### 1. Conversation 模型重构（重要）

| 字段 | v0.4 | v0.5 | 前端影响 |
|---|---|---|---|
| `conversations.project_id` | 存在 | **已删除** | 不能再按 project 过滤 conversation（用户每个项目不再有独立 conv） |
| `conversations.status` | 不存在 | 新增 `active/invalidated/archived` | UI 要支持"新建对话"按钮（invalidate 当前 active） |
| `conversations.summary` | 不存在 | 新增（v0.5 阶段 B 自动压缩） | UI 可选展示"对话历史摘要" |
| `conversations.message_count` | 不存在 | 新增 | UI 可展示"消息数" |
| `messages.project_id` | 不存在 | **新增**（nullable） | 后端自动填，前端读 chapter 时可按 project_id 过滤历史消息 |
| `messages` 自动 +1 `message_count` | 不存在 | 新增 | 每次消息自动 +1 |

**前端需要做**：
- ✅ 移除"每个项目独立对话"的假设
- ✅ 新增"新建对话"按钮（POST 后端会让旧 conv 变 invalidated）
- ✅ UI 可选展示 `status` 字段（active/invalidated/archived）

### 2. 章节 metadata 增强

`POST /api/projects/{id}/chapters` 返回多了 `summary` 字段（v0.5 自动生成 1 句 summary）：

```json
{
  "chapter_num": 5,
  "title": "第 5 章",
  "content": "...",
  "word_count": 3684,
  "generated_at": "2026-06-18T12:30:00",
  "new_seeds_triggered": 0,
  "summary": "林远在数据黑市遇到自称认识他失踪父亲的神秘卖家..."
}
```

**前端需要做**：
- ✅ Reader 页章节顶部展示 1 句 `summary`（胶囊样式）

### 3. 业务接口落库（影响前端 debug）

5 路由（project/chapter/action）调一次会自动写 `messages` 表 `role=tool` + `tool_calls/tool_results`。前端**无感**——只是管家 LLM 下次调用能"看到"你之前手动做的操作。

### 4. EditArtifactTool 新增（管家 LLM 解析对话用）

管家 LLM 调 `edit_artifact` 工具的结构化 diff 不会被前端直接看到（仍走 WS 流式回复），但**前端要支持**：
- ✅ 管家回复里如果有"我刚刚把主角妹妹改成了林雪" → 前端展示"应用到基座"按钮
- ✅ 用户点"应用" → 前端调 PATCH /api/projects/{id}/base 或 POST /api/projects/{id}/edit-artifact（新端点 v0.6）

> 注：v0.5 阶段 A 的 edit_artifact 工具**已经实现**，但**没有直接 HTTP 端点**——管家在 WS 里调。前端对接 v0.5 时暂时不接 edit_artifact，等 v0.6 加 HTTP 端点。

### 5. WebSocket 协议（暂未变）

WS 端点 `ws://host/api/chat` —— v0.4+ 没有大改，前端用旧协议即可。

## 14 个 REST 端点速查

### 项目（4 个）

#### `GET /api/projects` — 列出项目

```bash
curl "http://localhost:7778/api/projects?limit=20&offset=0"
```

**响应**：
```json
{
  "total": 3,
  "projects": [
    {
      "id": "test-proj",
      "name": "我的小说",
      "palette": "modern",
      "chapter_count": 5,
      "last_updated": null
    }
  ]
}
```

#### `POST /api/projects` — 创建项目

```bash
curl -X POST "http://localhost:7778/api/projects" \
  -H "Content-Type: application/json" \
  -d '{"name": "我的小说", "palette": "modern", "initial_prompt": "一个关于..."}'
```

**响应**：
```json
{
  "id": "test-proj",
  "name": "我的小说",
  "created_at": "",
  "onboarding_required": true
}
```

> 注意：v0.5 不返回 `created_at`（项目元数据在 DB，但没在 response 里）—— 后续 v0.6 补上。

#### `GET /api/projects/{id}` — 加载项目详情

```bash
curl "http://localhost:7778/api/projects/test-proj"
```

**响应**（v0.5 关键变化）：
```json
{
  "id": "test-proj",
  "name": "我的小说",
  "palette": "modern",
  "seven_artifacts": {
    "01-world-tree.yaml": { "base": {...}, "branches": [], "metadata": {} },
    "02-style-charter.yaml": {...},
    "03-genre-resonance.yaml": {...},
    "04-main-plot.yaml": {...},
    "05-sub-plot.yaml": {...},
    "06-character-card.yaml": {...},
    "07-seed-table.yaml": {...}
  },
  "world_tree": { "base": {...}, "branches": [], "metadata": {} },
  "chapters": [
    {
      "num": 1,
      "title": "第 1 章",
      "summary": "林远在数据黑市...",
      "word_count": 3684,
      "file_path": "data/projects/test-proj/chapters/chapter_001.md"
    }
  ]
}
```

> **v0.5 关键**：`chapters[].summary` 自动有值了（之前 v0.4 是 null）。`seven_artifacts` 是 7 件完整 dict（取代 v0.4 文件读取）。

#### `DELETE /api/projects/{id}?confirm=true` — 删除（软删）

```bash
curl -X DELETE "http://localhost:7778/api/projects/test-proj?confirm=true"
```

## WebSocket 协议（暂未变）

```typescript
// 连接
const ws = new WebSocket('ws://localhost:7777/api/chat');

// 发消息
ws.send(JSON.stringify({
  type: 'user_message',
  content: '我想要一个赛博朋克小说',
  project_id: 'test-proj',  // 可选
  conversation_id: 'xxx',   // 可选（v0.5: 一对一，server 端会忽略或覆盖）
}));

// 收消息
ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  // msg.type: 'assistant_chunk' | 'assistant_done' | 'tool_call' | 'error'
};
```

**v0.5 重要**：管家 LLM 自动有"多轮上下文"（从 messages 表拿 history），前端不用手动维护 conversation_id。

## 前端需要做的 3 个 v0.5 适配

### 1. Reader 页：显示章节 summary

章节顶部加 1 句 summary 胶囊：

```vue
<template>
  <div class="chapter-header">
    <div class="summary-capsule">{{ chapter.summary }}</div>
    <h1>{{ chapter.title }}</h1>
  </div>
</template>
```

### 2. 会话管理 UI：新建对话

```vue
<button @click="handleNewConversation">+ 新建对话</button>

<script setup>
async function handleNewConversation() {
  // 调后端 API 让旧 conv 失效
  // v0.5 后端: get_or_create_conversation 会自动 invalidate 旧的
  // 前端只需刷新页面 / 重连 WS
  await fetch('/api/conversations/new', { method: 'POST' });
  // 或重连 WS
  ws.close();
  ws = new WebSocket('ws://...');
}
</script>
```

> v0.5 阶段 A 没新增 `/api/conversations/new` HTTP 端点（仅 WS 内部支持）—— 蕾姆在 v0.6 阶段 A 补上。前端暂时用 WS 重建会话。

### 3. 管家回复解析

管家 WS 回复里如果有 `tool_call: edit_artifact` 类型，前端展示 diff：

```typescript
ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  if (msg.type === 'tool_call' && msg.tool_name === 'edit_artifact') {
    // 展示 diff 给用户确认
    showDiffModal(msg.args.target, msg.args.operation, msg.args.data);
  }
};
```

> v0.5 EditArtifactTool 已经在 specialist 里用，但**HTTP 端点暂未暴露**（WS 内部调用）—— 前端暂时**只读不写**。

## 常见问题

**Q: 我应该用 v0.4 旧 API 还是 v0.5 新 API？**
A: v0.5 全部向后兼容，5 路由 response 都加字段（不删字段），前端 v0.4 代码直接跑 v0.5 后端也 OK。**只需要补 Reader summary 展示 + 会话管理 UI**。

**Q: EditArtifactTool 怎么用？**
A: v0.5 阶段 A 已经在 specialist W2 真实实现里调了，但**没有 HTTP 端点**。前端暂不接，等 v0.6 加 HTTP 端点。

**Q: 我调 create_project 失败 409 怎么办？**
A: 项目 ID 已存在。换个名字。

**Q: 生成章节超时怎么办？**
A: 30-60s 正常。如果超过 120s 还没响应，前端 abort 并提示用户"章节生成中，请稍候"。
